01 · Context

The Problem

Understand this before you prompt your first line of code.

"Access to Education" is a non-profit organization dedicated to supporting underprivileged students through their academic journey. They currently manage student information across disconnected spreadsheets, paper files, and email threads.

The result: counselors lose track of student progress, scholarship deadlines are missed, mentor meetings go unscheduled, and critical intervention opportunities are overlooked. Students fall through the cracks when they need support most.

They need a unified web application where counselors can view comprehensive student profiles, track scholarship applications, monitor academic progress, and manage mentorship relationships. Every interaction must be logged so no student is forgotten.

02 · Screens

What to Build

Four interconnected modules. All must be functional, connected to the real API, and deployed live.

MODULE 01

Student Directory

Searchable, filterable list of all students. Each card shows name, photo, academic year, major, enrollment status, and quick stats (GPA, credits completed). Clicking opens the full profile.

MODULE 02

Student Profile Dashboard

Comprehensive view with four sections: Profile Header (photo, contact, demographics), Academic Progress (GPA trend, credits, current courses), Scholarship Tracker (applications with status and deadlines), and Mentorship Panel (assigned mentor, meeting history, notes).

MODULE 03

Scholarship Management

Interface to view all scholarship applications for a student. Shows provider, amount, deadline, status (Researching/Applied/Interview/Awarded/Rejected), and requirements. Ability to add new applications and update statuses.

MODULE 04

Mentorship & Meetings

View assigned mentor details and full meeting history. Schedule new meetings with date, duration, and notes. Chronological log of all mentorship interactions with searchable notes.

03 · Requirements

What's Expected

Core requirements must be met. Bonus items are genuinely optional — a solid core beats a buggy everything.

AI & Documentation
  • PROMPTS.md — Document your key AI prompts: initial scaffolding, data model decisions, dashboard component architecture, and debugging sessions. Show the evolution, not just final prompts.
  • AI_REVIEW.md — Critique your own codebase: 2 things the AI did well, 2 things you had to manually fix or rewrite, and 1 technical debt item you'd address next.
  • Decision audit — In your README, explicitly note which architectural decisions were AI-suggested vs. your overrides. We want to see your judgment, not just AI output.
Backend
  • REST API in Node.js / Express or Next.js API routes with consistent JSON responses throughout.
  • Data model covering Students (profile, demographics, academic info), Scholarships (applications with status tracking), Mentors (mentor profiles and assignments), and Meetings (mentorship session logs with timestamps).
  • Core endpoints: list/search students · get student with full profile · create/update student · manage scholarships · mentor assignment · meeting CRUD.
  • Data validation — ensure GPA is within valid range, enrollment status transitions are logical, and scholarship deadlines are tracked.
Frontend
  • React (preferred) with TypeScript, or Next.js 14+. No UI library required — how you write CSS is part of the evaluation.
  • All four modules must be functional and connected to the real API. No hardcoded mock data at submission.
  • Dashboard must include data visualization for academic progress (GPA trends, credit completion) using a charting library of your choice.
  • Deployment — Host your frontend on Vercel, Netlify, or similar. The API must be accessible (Render, Railway, Fly.io, etc.). Include live URLs in README.
Both
  • Loading and error states — the UI must show when data is loading and handle API errors gracefully. Silent failures are not acceptable.
  • Date & time tracking — every meeting, scholarship deadline, and status change records a precise timestamp.
  • Environment configuration — Your README must clearly document all env vars. We will test your deployment.
  • Accessibility — Keyboard navigation, screen reader support, and WCAG 2.1 AA color contrast compliance.
Bonus
  • Seed script with realistic dummy data (15+ students, 5+ mentors, 20+ scholarships) — makes the demo far more compelling.
  • Export functionality — allow exporting student reports or scholarship summaries as PDF or CSV.
  • Real-time updates — WebSocket or Server-Sent Events for live scholarship status updates.
  • Any unit or integration test — validation logic, API endpoints, anything that shows you thought about correctness.
04 · Data Model

Reference Schema

Use this as a starting point. You may extend or modify based on your architectural decisions.

Student Entity

{
  id: string;
  firstName: string;
  lastName: string;
  email: string;
  avatarUrl?: string;
  academicYear: 'Freshman' | 'Sophomore' | 'Junior' | 'Senior';
  major: string;
  gpa: number; // 0.0 - 4.0
  enrollmentStatus: 'Full-time' | 'Part-time' | 'Leave of Absence' | 'Graduated';
  creditsCompleted: number;
  creditsRequired: number;
  expectedGraduation: Date;
  demographics: {
    firstGeneration: boolean;
    lowIncome: boolean;
    underrepresentedMinority: boolean;
  };
  createdAt: Date;
  updatedAt: Date;
}

Scholarship Entity

{
  id: string;
  studentId: string; // Foreign key
  name: string;
  provider: string;
  amount: number;
  currency: string;
  status: 'Researching' | 'Applied' | 'Interview' | 'Awarded' | 'Rejected';
  deadline: Date;
  requirements: string[];
  essayRequired: boolean;
  essaySubmitted?: boolean;
  notes: string;
  dateApplied?: Date;
  createdAt: Date;
  updatedAt: Date;
}

Mentor & Meeting Entities

{
  // Mentor
  id: string;
  name: string;
  title: string;
  company: string;
  expertise: string[];
  email: string;
  bio: string;
  maxMentees: number;

  // Meeting
  id: string;
  studentId: string;
  mentorId: string;
  date: Date;
  duration: number; // minutes
  notes: string;
  actionItems: string[];
  status: 'Scheduled' | 'Completed' | 'Cancelled';
  createdAt: Date;
}
